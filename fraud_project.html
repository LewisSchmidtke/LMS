Synthetic Data Generation
Four generator classes (User, Device, PaymentMethod, Merchant) create realistic entities using weighted distributions via Faker. TransactionGenerator produces both normal and fraudulent chronological patterns.

Technical Implementation

Core transaction pattern generation logic — the heart of the training data pipeline. Each pattern is anchored to a TransactionContext dataclass carrying all static fields, with dynamic fields filled during generation. For full source, visit GitHub.

src/TransactionGenerator.py
from dataclasses import dataclass
import random
from datetime import datetime, timedelta

@dataclass
class TransactionContext:
    user_id: int
    device_id: int
    merchant_id: int
    currency: str
    is_fraudulent: int
    fraud_type: str | None
    payment_id: int | None
    transaction_status: str | None

class TransactionGenerator:

    # ── Card Probing: many small, mostly-declined hits ──────────────
    def _card_probing_pattern(self, context):
        # 3–15 transactions, 30–90s apart, 10% approval rate
        pattern = []
        ts = datetime.now()
        for _ in range(random.randint(3, 15)):
            context.transaction_status = 'approved' if random.random() < 0.10 else 'declined'
            ts += timedelta(seconds=random.uniform(30, 90))
            pattern.append(self._build_transaction(context, self._small_amount(), ts))
        return pattern

    # ── Botting: rapid repeat purchases, same payment method ────────
    def _botting_pattern(self, context):
        # 1–10 transactions, 0.5–1s apart, 90% approval rate
        pattern = []
        ts = datetime.now()
        for _ in range(random.randint(1, 10)):
            context.transaction_status = 'approved' if random.random() < 0.90 else 'declined'
            ts += timedelta(seconds=random.uniform(0.5, 1.0))
            pattern.append(self._build_transaction(context, self._generate_amount(), ts))
        return pattern

    # ── Normal: 1–3 organic purchases with natural timing ───────────
    def _normal_pattern(self, context):
        pattern = []
        ts = datetime.now()
        for _ in range(random.randint(1, 3)):
            context.transaction_status = self._determine_status(is_fraud=False)
            ts += timedelta(seconds=random.uniform(300, 86400))
            pattern.append(self._build_transaction(context, self._generate_amount_by_distribution(), ts))
        return pattern

Challenges & Solutions

Building a production-grade fraud detection system from the ground up raised several non-trivial engineering problems.

Imbalanced Datasets
Fraud represents a small fraction of real-world transactions, causing models to bias toward the majority class. Both XGBoost (scale_pos_weight) and Random Forest (class_weight="balanced") support direct weighting, and SMOTE oversampling is available as an alternative for either model.
Realistic Fraud Pattern Diversity
Fraudulent transactions span wildly different behavioural signatures — rapid micro-transactions in Card Probing versus slow account takeovers. Each pattern requires its own timing distributions, amount clusters, and approval logic to be useful as training signal, rather than noise.
Real-Time Latency Requirements
Fraud scoring must complete within the transaction approval window. This drove the choice of ONNX export and NVIDIA Triton, which provides model versioning, batching, and sub-millisecond inference — currently planned once model training stabilises.
Scalable Feature Engineering
Computing windowed aggregates (e.g. decline rate in the last hour, unique merchants in 24h) across millions of transactions demands distributed processing. Apache Spark handles this via PySpark window functions, with a Spark Master + Worker cluster defined in docker-compose.yml.
